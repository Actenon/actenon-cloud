# Pilot Packaging

## Purpose

This document defines the recommended early packaging model for Actenon Cloud in design-partner pilot conversations.

It is specific to the invoice payment execution wedge. It does not define mature enterprise packaging or broad future SKUs.

## Packaging Recommendation

The strongest early packaging model is:

- a hybrid commercial model expressed in two phases
- a fixed-scope, fixed pilot or setup fee for the current design-partner motion
- followed, if the pilot succeeds, by live operational pricing tied to proved and allowed consequential actions
- with blocked actions free from usage charging

This means the commercial packaging should be split into two distinct phases:

1. pilot packaging now
2. live operational packaging later

The pilot should not be priced as if it were already steady-state production software.

## Why Pilot Packaging Must Be Separate

The pilot creates value through a mix of:

- product capability
- implementation and configuration work
- workflow mapping
- seeded validation
- hands-on support
- explicit review and sign-off

That is not the same value shape as later live operational use.

Trying to price the pilot as a pure recurring subscription or pure per-action meter would be harder to explain and harder for a design partner to trust.

## Recommended Pilot Package

### Base Pilot Package

Recommended unit:

- one tenant
- one invoice payment workflow family
- one fixed time box
- one agreed success plan

Recommended structure:

- fixed pilot or setup fee
- fixed pilot scope
- fixed delivery window

Suggested duration:

- `6 to 8 weeks`

### What The Base Pilot Package Includes

- environment and access setup support
- tenant and workflow configuration support
- policy setup and workflow rule mapping
- approval and evidence workflow enablement
- proof issuance enablement
- escrow lifecycle enablement
- receipt ingestion and audit trace review
- seeded validation scenarios
- controlled pilot review sessions
- final findings review and recommendation

### What Is Separately Scoped

- customer-specific adapter or connector work
- pilot extension beyond the agreed time box
- additional workflow families
- production-hardening work
- managed signing rollout
- real capability-release integration
- enterprise auth rollout

## Recommended Live Operational Packaging Direction

If the pilot succeeds and the partner wants to continue toward live operational use, the strongest model to test is:

- usage-based pricing for proved and allowed consequential actions
- with setup, rollout, and integration work still scoped separately when needed

This is the best next-step packaging direction because:

- value is still strongly tied to consequential actions that are actually allowed and progressed
- buyers can understand paying for real governed actions more easily than paying for abstract software access
- the setup work has already been handled explicitly through the fixed pilot or through separate scoped work, rather than hidden inside a recurring platform fee

## What Not To Package Yet

Do not introduce:

- multiple enterprise editions
- broad role-based or module-based SKU design
- complex action-family bundles
- speculative verifier licensing bundles

Those would outrun current product maturity and current commercial evidence.

## Best Packaging Summary

For current pilot conversations, the cleanest packaging story is:

- use a fixed pilot or setup fee for a tightly scoped pilot
- scope extra integration separately
- if the pilot proves value, move to usage pricing tied to proved and allowed consequential actions
- keep blocked actions free from usage charging

That is the most commercially legible fit for the repo's current maturity.
