# Pilot Scope And Pricing

## Purpose

This document defines the commercial scope and pricing structure for the invoice payment execution governance pilot.

It intentionally avoids fake list pricing. The current repo does not establish approved public pricing numbers. Instead, this document defines the offer shape and pricing logic that should be completed during deal approval.

## Recommended Commercial Model

The recommended commercial model is:

- a paid pilot
- fixed scope
- fixed time box
- one invoice payment governance wedge
- optional separately scoped integration services if needed

Suggested pilot duration:

- `6 to 8 weeks`

Suggested fixed-fee pilot structure:

- one fixed fee for the base pilot
- one agreed scope for one tenant and one invoice payment workflow family
- optional separately scoped add-ons for customer-specific integration work
- no public list pricing in the repo; fee approved per deal

This is the strongest commercial structure for the current repo because:

- the product is real enough for a serious pilot
- the scope is intentionally narrow
- production hardening is still a separate later decision

## Included Pilot Scope

Included by default:

- one tenant pilot
- one invoice payment workflow family
- one suggested `6 to 8 week` engagement window
- pilot configuration support
- policy setup guidance
- approval and evidence workflow enablement
- proof issuance and escrow lifecycle enablement
- receipt ingestion and audit trace review
- seeded validation scenarios
- business-hours pilot support
- final findings review

Customer responsibilities within the base pilot:

- provide named pilot sponsor, technical owner, and finance operations owner
- provide requester, approver, and release-manager users
- provide invoice payment proposal source and sample scenarios
- provide policy thresholds and approval rules
- provide receipt source and downstream execution process
- provide controlled pilot environment inputs and security approvals

## Optional Add-Ons

Optional, separately scoped items:

- customer-specific shallow adapter work
- custom reporting or export formatting
- pilot extension beyond the original time box
- additional workflow variants beyond the agreed invoice payment scope

## Out Of Scope For Pricing

Not included in the base pilot offer:

- production deployment hardening
- enterprise identity rollout
- managed signing rollout
- real external capability release rollout
- multi-workflow expansion
- broad UI work

## Pricing Structure

The pricing structure should be completed with approved commercial terms for each customer.

Recommended fields:

- base pilot fee: `[deal-specific amount]`
- optional integration services fee: `[deal-specific amount]`
- optional extension fee: `[deal-specific amount]`
- billing cadence: `[upfront / milestone / mixed]`

Suggested structure for commercial approval:

- base fixed fee covers the agreed pilot scope
- optional add-ons cover extra integration or pilot extension work
- post-pilot production-hardening work is priced separately

## Packaging Units

The pilot should be packaged commercially around:

- one tenant
- one workflow family
- one time-boxed engagement
- one named support model
- one agreed success plan

This keeps the commercial motion aligned to the current product reality.

## Why There Is No Public Price Here

The repo cannot honestly publish a fixed public pilot price today because pricing depends on:

- pilot duration
- customer environment complexity
- receipt integration expectations
- support intensity
- optional shallow adapter work

The honest commercial posture is proposal-based pricing during the current pilot stage.

## Post-Pilot Commercial Path

If the pilot succeeds, the next commercial step should be a separately scoped production-hardening engagement or paid control-plane deployment plan.

That next step should price separately for:

- identity hardening
- signing hardening
- real capability release
- deployment and observability hardening
- broader workflow rollout

Possible post-pilot expansion path:

- expand from one invoice payment workflow to additional payment-control workflows
- deepen integration from simulated release to real protected-resource release
- harden identity, signing, and deployment posture for production readiness
