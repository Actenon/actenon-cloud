# Pilot GTM Gap Analysis

## Purpose

This document identifies the remaining GTM gaps for the invoice payment execution pilot based on the current repo surface.

It assumes the repo already has:

- a technically credible product slice
- a design-partner pilot package
- a one-pager
- proposal and SOW templates
- a buyer FAQ
- a pilot trust summary

## Current GTM Position

### What Exists Now

- a credible finance-control wedge
- a clear open kernel versus paid control-plane story
- a commercially usable pilot package
- proposal and SOW scaffolding
- a pilot scope and pricing framework
- honest risk, limitation, and readiness documents

### What Does Not Exist Yet

- ideal customer profile definition
- buyer persona framing
- approved public or repeatable pricing
- ROI narrative
- open-kernel versus paid-control-plane packaging matrix
- pilot qualification criteria
- final pilot readout and conversion playbook

## GTM Readiness By Motion Stage

### 1. Outreach

- there is now a real one-pager and a real invoice payment execution wedge

Current gap:

- no concise outreach message tailored separately for finance operations and technical buyers
- no ICP document
- no documented “why now” narrative beyond the technical pilot story

What is still needed:

- buyer persona memo
- ICP and qualification criteria

### 2. Discovery And Qualification

Current strength:

- no clear “not a fit” guidance

What is still needed:

- design-partner qualification rubric

### 3. Solution Framing

Current strength:

- the invoice payment workflow is narrow and understandable
- the open kernel and paid control plane are clearly separated

Current gap:

- the buyer-facing language is still more operational than executive
- there is no “before versus after” comparison for the finance workflow
- there is no concise articulation of why the paid layer is worth buying above the open kernel

What is still needed:

- open-kernel versus paid-control-plane matrix
- buyer-facing workflow transformation note

### 4. Proposal And Pilot Closing

Current strength:

- the repo now contains a proposal template, a SOW template, and a scope-and-pricing framework

Current gap:

- pricing is still a framework, not an approved offer model
- there is no standard time-box guidance strong enough for repeatable selling
- there is no mutual success plan template

What is still needed:

- repeatable pilot duration guidance
- mutual success plan
- pricing approval path

### 5. Pilot Execution

Current strength:

- operator journey

Current gap:

- no explicit pilot governance cadence
- no final findings-readout template

What is still needed:

- final readout template

### 6. Conversion To Paid Control Plane

Current strength:

Current gap:

- no commercial packaging that explains when an open-kernel user should buy the paid layer
- no pricing posture beyond the pilot framework
- no post-pilot conversion memo
- no ROI support for the commercial decision

What is still needed:

- open-kernel versus paid-control-plane packaging and conversion guide
- ROI framing for invoice payment execution

## Biggest GTM Gaps

### Gap 1: No ICP Or Buyer Persona Layer

The biggest immediate GTM weakness is that the materials are still broad and document-heavy.

They do not yet identify clearly:

- primary buyer
- technical buyer
- executive sponsor
- pilot champion
- day-to-day operator

### Gap 2: No Repeatable Qualification Model

The repo still lacks a simple answer to:

- who should run this pilot
- who should not
- what minimum customer maturity is required

### Gap 3: No Approved Packaging And Pricing Story

`PILOT_SCOPE_AND_PRICING.md` is honest and useful, but it is still only a framework.

The repo still does not have:

- approved pricing
- packaging tiers
- entitlement boundaries

### Gap 4: Weak ROI And Budget Justification

The metrics are good, but there is still no short commercial narrative for:

- avoided wrong-payee or wrong-amount payments
- reduced approval ambiguity
- improved audit-readiness
- time to safe execution decision

The repo needs ROI language without inventing results.

### Gap 5: Conversion From Open Kernel To Paid Control Plane Is Not Yet Commercially Tight

The technical boundary is clear. The GTM boundary is not yet sharp enough.

What still needs to be explained better:

- why the open kernel is not enough for a governed invoice payment workflow
- what the paid control plane adds commercially
- what event triggers the move from open usage to paid control-plane deployment

## What Is Missing For A Real Pilot Proposal

The repo is now much closer than before, but it still needs the following for a fully mature pilot proposal motion:

1. an ICP and qualification document
2. buyer persona framing
3. a mutual success plan
4. standard pilot duration and volume guidance
5. a final readout template
6. an ROI narrative
7. an open-kernel versus paid-control-plane packaging matrix
8. a post-pilot conversion memo
9. a standard executive summary for proposals
10. an internal pricing approval path

## Open Kernel To Paid Control Plane Conversion Gaps

The repo already defines the technical split:

- open kernel: public contracts, execution semantics, verifier logic
- paid control plane: invoice payment governance, approvals, evidence, proof orchestration, escrow lifecycle, receipts, audit, tenancy, and admin controls

The missing commercial articulation is:

- when the open kernel alone is enough
- when a customer needs the paid control plane
- how invoice payment execution governance becomes the first commercial upgrade

This is the most important open-kernel versus paid-control-plane GTM gap after pricing and buyer targeting.

The repo is now materially stronger for real pilot proposals than it was before.

The next GTM work should not be new product design. It should be:

- targeting
- qualification
- ROI framing
- conversion packaging

That is the main path from a technically credible pilot to a commercially repeatable one.
