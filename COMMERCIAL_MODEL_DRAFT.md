# Commercial Model Draft

## Purpose

This document compares the leading commercial model options for Actenon Cloud and recommends the strongest early model for design-partner pilot conversations.

It is scoped to the invoice payment execution wedge.

It is not final pricing policy.

## Evaluation Standard

The commercial model should match:

- actual value delivered
- buyer trust and incentive alignment
- pilot delivery reality
- ease of explanation
- ease of saying yes in an early design-partner conversation

## Candidate Model 1: Fixed Fee For Pilot Or Platform Access

### Description

Charge a recurring or flat access-style fee for the pilot environment and software.

### Strengths

- simple to explain
- predictable budget for the customer
- matches the fact that the pilot includes meaningful fixed work

### Weaknesses

- too abstract if used as the only long-term commercial logic
- does not reflect the action-linked nature of the value in this wedge
- can sound like generic software access rather than governed consequence-linked control

### Verdict

Strong for the pilot itself.

Weak as the only long-term commercial logic.

## Candidate Model 2: Usage-Based Pricing Per Proved And Allowed Action

### Description

Charge only when a governed invoice payment is proved, allowed, and progressed as a consequential action.

### Strengths

- highly aligned to visible action-level value
- easy to connect to consequential business events
- can feel fair to buyers who want to pay for real governed outcomes

### Weaknesses

- weak fit for the pilot stage because delivery effort is front-loaded
- low pilot volumes would make pricing fragile
- ignores the ongoing fixed value of governance, audit, workflow state, and readiness surfaces
- can underprice customers with low volume but high governance complexity

### Verdict

Not the strongest model for the pilot stage by itself.

Potentially useful later as part of a broader commercial model.

## Candidate Model 3: Hybrid Fixed Pilot Or Setup Fee Plus Usage-Based Pricing For Proved And Allowed Actions

### Description

Charge:

- a fixed pilot or setup fee for implementation and readiness work
- plus usage-based pricing for proved and allowed consequential actions

### Strengths

- matches both setup value and live operational value
- easier to explain than a large SKU system
- aligns pricing to real governed actions without pretending the product is generic software access
- fits the trust model better than charging for every intake event

### Weaknesses

- requires disciplined explanation so the setup fee and the usage fee do not blur together
- requires operational clarity on what counts as a billable governed action
- still needs founder judgment on exact usage definitions and rollout mechanics

### Verdict

This is the strongest overall commercial model for the current repo state.

## Candidate Model 4: No Fee For Blocked, Denied, Or Prevented Actions

### Description

Do not charge usage for:

- blocked actions
- denied actions
- structurally non-executable actions
- prevented actions that do not progress as governed allowed actions

### Strengths

- strongest trust alignment
- avoids the appearance that the vendor profits from rejection volume
- reinforces the product's role as a safety and control layer
- easy to explain in pilot conversations

### Weaknesses

- blocked events still create real value and operating load
- some future customers may argue that prevention itself should count as billable value

### Verdict

Recommended.

Blocked or denied actions should generally be free from a usage-pricing standpoint.

Their value should support the case for the platform, not be monetized as a separate action fee.

## Recommended Commercial Model

### Recommendation For The Current Pilot Stage

The strongest pilot commercial model is:

- fixed pilot or setup fee with fixed scope
- optional separately scoped implementation or integration services
- no per-action usage billing during the pilot

This is the best fit because the pilot is sold as:

- a controlled evaluation
- a narrow workflow deployment
- a time-boxed delivery engagement
- a trust-building exercise around a critical finance workflow

### Recommendation For Later Live Operational Use

If the pilot proves value and the partner wants to continue, the strongest commercial model to test is:

- hybrid setup fee plus usage-based pricing for proved and allowed actions

This is stronger than pure recurring subscription because the wedge creates visible action-linked value.

It is stronger than pure per-action usage alone because important delivery and rollout work still needs explicit commercial treatment rather than being hidden in the action meter.

## Why This Model Is Strongest

### 1. It Matches Delivery Reality

The current pilot includes substantial fixed work:

- workflow setup
- control mapping
- validation
- support
- review and sign-off

That makes a fixed pilot fee the cleanest structure now.

More precisely, it makes a fixed pilot or setup fee the cleanest structure now.

### 2. It Matches Buyer Psychology

Buyers are more likely to say yes to:

- one clear pilot fee
- one clear scope
- one clear time box

and then a simple action-linked model later, rather than a generic software subscription during the early motion.

### 3. It Preserves Trust

Charging later for proved and allowed actions is easier to justify than charging for every attempt.

Not charging for blocked actions helps preserve trust and incentive alignment.

### 4. It Leaves Room For Real Pricing Learning

The repo is not mature enough to pretend the long-term commercial model is already finalized.

This draft preserves flexibility while still recommending a strong early path.

## What Is Charged Now Versus Later

### Charged Now

- fixed pilot or setup fee
- optional separately scoped integration or extension work

### Not Charged As Usage During The Pilot

- individual intake requests
- blocked actions
- denied actions
- prevented actions

### Possible Later Live Charges To Test

- per proved and allowed action

## What Remains Out Of Scope

This draft does not define pricing for:

- production hardening
- enterprise auth rollout
- managed signing rollout
- real capability-release rollout
- broad workflow expansion
- external verifier licensing or commercial terms

Those should be treated separately if and when the commercial motion reaches that stage.

## Main Commercial Risks

- buyers may still push for pure consequence-based pricing without wanting to pay for setup or rollout work
- a hybrid model could feel too complex if explained poorly
- low-volume pilots may make action-based economics look less compelling
- customers may undervalue blocked-action prevention if it is not translated clearly into business terms
- pilot success may create pressure to jump to production expectations before the platform is ready

## Best Draft Summary

The strongest early answer is:

- fixed pilot or setup fee now
- optional scoped services for extra work
- blocked actions free from usage charging
- usage pricing tied to proved and allowed actions after the pilot, with setup or rollout work still scoped separately when needed

That is the most commercially effective model for the current product, buyer psychology, and pilot maturity.
